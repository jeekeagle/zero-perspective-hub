程序视角

把提示词当作"源码",看它如何获得类型、模块、版本这些程序性特征

编程 DSL Prompt

一问题的提出

很多人对"prompt 是程序"这个说法嗤之以鼻。我理解, 但我越来越觉得,这是真的 —— 只是它运行在一台你看不见的机器上。

戴"程序"这副眼镜看 prompt,我们能立刻发现它和传统程序的异同。 同的部分可以借鉴,异的部分必须新造。

二它越来越像程序

早期 prompt 是一段自然语言描述:"请你扮演一个友好的客服,回答用户的提问。" 那不是程序,那是一段说明文。

现在的 prompt 长这样:

# role
你是 {{ company_name }} 的高级客服。

# context
- 已知产品: {{ product_list }}
- 当前用户: {{ user_segment }}
- 上一轮已答: {{ previous_turn }}

# task
判断用户意图 → 选择工具 → 输出 JSON 格式响应

# output_schema
{
  "intent": "refund | track | info | handoff",
  "tool": "string | null",
  "reply": "string",
  "confidence": "0-1"
}

这不就是带模板的 DSL 吗?

三三个相似之处

  • 类型系统 —— 通过 output_schema 强制模型输出结构化数据,这是运行时类型检查。
  • 模块化 —— 大 prompt 拆成小 prompt 拼装,这是函数组合。
  • 版本管理 —— 同一个 skill 不同 prompt 版本的 diff,这是语义化版本。

四三个不一样的地方

  • 编译是模糊的 —— 同一个 prompt 跑十次,得到十次略有不同的结果。
  • 依赖是隐式的 —— 你的 prompt 强依赖于某个模型版本,但这层依赖不在你的代码里,而在模型权重里。
  • 调试是叙事性的 —— 你不能"打断点",你只能反复"问它当时在想什么"。

这三个"不一样"不是缺陷,而是这种新程序的本质特征。 试图用传统程序的标准去评价它,会错过它真正擅长的场景。

五一个结论

我不再把 prompt 当作文本。我把它当作源码。 源码就该有:版本控制、单元测试、code review、重构历史。

那些只把 prompt 当作"问 AI 的句子"的人, 就像 2005 年还在用 HTML 写网站 —— 不是不能,但效率差一个数量级。

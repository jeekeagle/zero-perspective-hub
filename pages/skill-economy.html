Skill 的经济学

2026-05-28 · 约 5 分钟读完 · Skill

如果你 2024 年写过 prompt,2025 年大概率也写过。 但"把 prompt 当作可交付物"这个心智,大部分人是 2025 年下半年才养成的。

从一次性 prompt 到 Skill

一个 prompt 从"我这次问 AI 用的句子",变成"我下次还要用、别人也可能用、值得维护"的东西, 这中间发生了什么?

三件事:

  • 模型能力稳定了 —— 同一个 prompt 重复跑,结果差异在可接受范围内。
  • 任务模板化了 —— 我们开始发现,80% 的 prompt 其实是同一类任务的变体。
  • 工具链出现了 —— 有 Git 一样的东西让我们管理这些 prompt,能版本化、能分享、能组合。

微技能市场

接下来的事情很像早期 npm:有人写了一个好用的"翻译技能"放在 GitHub, 有人 fork 它改成"翻译 + 校对",又有人把几个小技能拼成"翻译工作流"。 一个二级的 prompt 市场就这样长出来了,只不过交易的不是代码,是认知流程。

区别在于:

  • 代码是确定的,prompt 不是 —— "运行结果"取决于模型本身。
  • 代码可以被测试,prompt 很难 —— 你只能让别人再跑一遍,然后祈祷结果一致。
  • 代码有依赖,prompt 有"反依赖" —— 你写的 prompt 强依赖于某个模型版本,模型一升级,prompt 可能就崩了。

一个新的分工

我看到的分工是这样:一类人写通用技能(写邮件、改 bug、生成测试), 一类人写领域技能(法律审合同、医疗读片子、教学出题), 一类人写元技能(把上面两类组合起来,带状态机、带记忆、带纠错)。

第三类人最稀缺,也最像 90 年代的"系统架构师"—— 他们不写具体技能,他们让一堆技能能互相说话。

一个未解的问题

当你的"工作流" 100% 由 prompt 组成时,谁拥有这个工作流的产权? 是写 prompt 的人,是训模型的公司,还是"发现这个组合特别有效"的那位中间人?

我不知道答案。但我赌: 未来会有"Skill License",就像现在的 MIT / Apache 一样, 只是它授权的不是代码,而是某种认知工艺。

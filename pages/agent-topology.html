Agent 协作的拓扑学

2026-04-22 · 约 5 分钟读完 · Agent

2024 年,大家还在比谁的单体 Agent 更强。 2025 年,所有人都在画图。

三种基本拓扑

1. 串行(流水线)

A → B → C → D。每个 Agent 干一件事,把结果丢给下一个。 像工厂流水线,效率高,但一个卡住全卡住。

适用:流程清晰、步骤可拆的任务 —— 写代码、生成报告、做多轮审核。

2. 并行(投票 / 竞争)

同一个任务给 N 个 Agent 同时做,然后选最好的,或者让它们互相打分。 像陪审团,质量高,但费钱。

适用:质量比成本重要的场景 —— 关键决策、复杂推理、需要多种视角的问题。

3. 监督(分层)

一个"老板 Agent"指挥一群"员工 Agent",老板负责拆任务、检查结果、决定重做。 像公司,灵活但需要好的老板。

适用:复杂、开放、目标不明确的任务 —— 科研、长期项目、用户陪伴。

实际系统是混合的

真实生产里几乎没有人只用一种拓扑。 我自己跑过的几个项目,大致是: 

用户输入
  └─ 监督层(老板 Agent)
      ├─ 串行子流 1(规划 → 搜索 → 整理)
      ├─ 并行子流 2(三个审稿人投票)
      └─ 监督层回收 → 输出

也就是说,我们其实在画组织架构图。 老板在分活、员工在执行、HR 在做 review —— 全套人类组织语言都搬过来了。

一个有意思的观察

我做了一些小规模实验,发现:

选哪种拓扑,和任务的模糊度高度相关。 越模糊的任务,越需要监督层;越清晰的任务,越应该串行。 并行适合关键路径,不适合日常。

这其实和人类组织一样:流水线工人不需要老板,研究员必须有导师。 我们在重新发明组织管理学 —— 只不过这次的对象不是人,是 AI。

一个判断

我赌:未来一年会出现"Agent 拓扑描述语言",类似今天的工作流引擎(BPMN), 但语义是 AI-native 的。我们会开始争论: "你这个组织设计有反模式" —— 像今天 code review 一样。
